How fast does a Grace Blackwell GPU chew through unsalted SHA-1?

A short, hands-on investigation: stage mask attacks of increasing keyspace against three short synthetic passwords, measure where the wall is, and watch the cost curve.

When a service gets breached, the leaked database usually contains password hashes, not the passwords themselves — a hash is a one-way function: easy to compute forward, infeasible to reverse directly. But anyone holding the hashes can guess passwords offline at GPU speed and check each guess against the leak. Whether that takes seconds or geological time depends entirely on how the passwords were hashed. This is a look at what "GPU speed" actually means in 2026, against the weakest end of that spectrum: a single round of bare unsalted SHA-1.

The targets

Three synthetic 8-character passwords, hashed with a single round of bare unsalted SHA-1 (-m 100). They span the realistic surface that a hashcat mask attack can reach: pure lowercase + digits, mixed case, and symbol-flanked.

One fell. The other two survived this round — not because the hash is strong, but because the keyspace their pattern lives in is much bigger.

Methodology

A mask attack tells hashcat the shape of the password, not the password itself, then enumerates every candidate that fits that shape. The shapes are stacked cheapest-first:

  1. Lowercase + digits, lengths 6–9. Charset of 36, keyspace grows as 36L. Catches things like tabletop or 2faster.
  2. Upper-prefixed — one capital up front, rest lowercase+digit. Catches the Password1-style pattern.
  3. Symbol-flanked — uppercase/lowercase/symbol at the edges, lowercase+digit in the middle. The kind of "complex" password humans actually generate when forced to use symbols.

This ordering matters: each step multiplies the keyspace by something like 30–100, so the cost grows fast. You burn the cheap masks first because they take seconds, and you only commit GPU-days to the expensive ones if you have to.

The 8-character lowercase-and-digits mask is where the first crack landed: tabletop, a dictionary word. It took 55.3m. The next step in that family — length 9 with the same charset — would have taken roughly a day and a half. At that point the right answer isn't "throw more hashcat at it," it's "this mask family is wrong."

The cost curve

Same charset (lowercase + digits), one extra character at a time. Each step multiplies the keyspace by 36, so on a log axis it's a straight line. The teal bars are measured; the gray bars are projected at the GB10's observed peak rate of 857 MH/s.

Takeaway. Picking symbols doesn't save you. Length saves you. Going from 8 to 12 lowercase-and-digit characters — same boring charset, just longer — moves "an hour on a beefy GPU" to "longer than anyone reading this will be alive." And that's before salting, before a slow KDF, before any of the things a real password store should be doing.

Hardware

The GPU is an NVIDIA GB10 (Grace Blackwell) — NVIDIA's 2025 ARM-based workstation chip with unified CPU/GPU memory. Hashcat doesn't ship an ARM64 binary, so the setup script in the repo builds 7.1.2 from source against the system CUDA toolkit. Across the sweep it sustained ~857 MH/s on bare SHA-1.

Reproduce

git clone https://github.com/scheuclu/hash_cracking
cd hash_cracking
bash setup.sh                                      # builds hashcat 7.1.2 from source
export HASHCAT_BIN=$HOME/hashcat-7.1.2/hashcat
uv sync
uv run hash-cracking

Results stream to results.csv and per-config logs to runs/. Re-running python build_site.py regenerates this page from the CSV.
